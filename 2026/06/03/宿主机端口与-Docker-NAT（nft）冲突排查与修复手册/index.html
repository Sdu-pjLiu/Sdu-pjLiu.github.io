宿主机端口与 Docker NAT（nft）冲突排查与修复手册

发表于 2026-06-03 更新于 2026-06-25 分类于 计算机

适用场景：docker应用监听某端口，本机 curl 127.0.0.1:端口 正常，但局域网其他机器无法访问（或本机通过局域网 IP 自访也超时）。Docker 为 Snap 安装。

一、问题现象

现象 说明
本机回环正常 curl http://127.0.0.1:8008 返回正常
局域网 IP 访问失败 curl http://192.168.2.119:8008 超时（包括服务器自身）
抓包特征 enp5s0 上 SYN 包目标 IP 被改为 172.19.0.x（旧容器 IP）
docker ps PORTS 列 host 网络容器不显示端口映射，属正常
核心陷阱 iptables -t nat -S 看不到劫持规则，但 nft list ruleset 中清晰可见

根因：Snap Docker 通过 nftables 注入 DNAT 规则到内核。当 Docker 网桥被删除后，nft 规则未自动清理，持续将宿主机端口流量劫持到已不存在的容器 IP。

二、分层排查步骤

第一步：确认容器网络模式与监听

若仅 127.0.0.1 通、局域网 IP 不通（包括服务器自身），优先怀疑 NAT/DNAT 劫持。

第二步：检查 iptables（可能看不到问题）

关键陷阱：Snap Docker 通过 nftables netlink 直接注入内核规则，iptables-legacy 完全看不到。

第三步：确认 iptables 后端类型

系统同时存在 iptables-legacy 和 iptables-nft 两种后端。若当前为 legacy，Docker 通过 nftables 注入的规则不会被 iptables -S 显示。

五、常见问题

Q：为什么 iptables -t nat -S 看不到劫持规则？
A：Snap Docker 通过 nftables netlink API 直接向内核注入规则。若系统使用 iptables-legacy 后端，这些规则不在 iptables 视图内。必须用 nft list ruleset 查看。

Q：nft 命令不存在怎么办？
A：apt install nftables -y 即可。nftables 是用户态工具，内核中的 nft 规则已存在，安装后立即可查看。

Q：Snap Docker 自带的 nft 能用吗？
A：/snap/docker/3505/usr/sbin/nft 存在，但依赖 Snap 内部的 glibc 和 libnftables，直接运行通常会报库版本错误。直接装系统 nftables 更可靠。

Q：删除规则后重启 Docker 会恢复吗？
A：不会。这些规则是旧 Docker 网桥遗留的，当前 Docker daemon 不会重建已删除网络的规则。

Q：docker ps 看不到端口映射，是否代表 Docker 没参与？
A：不代表。host 网络不显示 PORTS，但 nftables NAT 表仍可能对宿主机端口做 DNAT。
