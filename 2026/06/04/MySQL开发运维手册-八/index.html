十九、查漏补缺：基础与资深运维清单

本节汇总正文未展开或仅一笔带过的 基础必知 与 资深运维 要点，便于自检「还缺什么」。

19.1 常见错误码与排查

错误码 含义 常见原因 处理方向
1045 Access denied 用户/密码/host 不匹配 查 mysql.user、授权、host 是否为 % / 实际 IP
2002 Can’t connect 服务未启、端口错、防火墙 systemctl status mysql、ss -tlnp | grep 3306
2003 Connection refused 未监听或 bind-address 查 bind_address、安全组
1040 Too many connections 连接打满 调 max_connections、应用连接池、查 sleep 连接
1205 Lock wait timeout 等锁超时 查锁等待、长事务、KILL 阻塞会话
1213 Deadlock 死锁 应用重试；优化加锁顺序
1227 Access denied (SUPER) 权限不足 用有权限账号或 REVOKE/GRANT 调整
3678 Schema 不一致 (8.0 GTID) GTID 集合冲突 查 gtid_executed / 复制状态，通常需 重建从库；慎 RESET MASTER / RESET BINARY LOGS（8.4+）

19.2 连接池、超时与安全

应用侧连接池（HikariCP、Django DATABASES 等）控制 最大连接数、空闲回收；避免每个 HTTP 请求新建连接。池大小 × 实例数 不应远超 MySQL max_connections 的 70%～80%。

常用超时参数：

参数 作用
wait_timeout 非交互连接空闲多久断开
interactive_timeout 交互式 mysql 客户端
net_read_timeout 读包超时
net_write_timeout 写包超时

安全加固清单：

  • 最小权限：业务账号仅 SELECT/INSERT/UPDATE/DELETE，禁止 SUPER、FILE。
  • 禁止应用使用 root；只读报表用只读账号（见 第十三节）。
  • 生产启用 SSL（若跨公网）：require_secure_transport=ON。
  • 使用 Prepared Statement 防 SQL 注入；禁止拼接 SQL。
  • 定期轮换密码；8.0 优先 caching_sha2_password 或按策略兼容旧客户端。

19.3 生产环境参数调优参考

以下为 单机 InnoDB OLTP 常见起点，需结合内存、磁盘、业务压测调整：

参数 建议方向 说明
innodb_buffer_pool_size 物理内存 50%～70% 过大导致 OS swap
innodb_buffer_pool_instances 8.0 常 1 或按 CPU 减少争用
innodb_log_file_size / redo 足够承接写峰值 8.0 见 #innodb_redo
innodb_flush_log_at_trx_commit 1=最安全；2=性能换持久性 主库一般 1
sync_binlog 1=最安全 与半同步配合
max_connections 按连接池总量 + 管理预留 过大增加内存
table_open_cache 表多时适当增大 避免频繁 open/close
tmp_table_size / max_heap_table_size 适度 过大内存临时表占 RAM
slow_query_log + long_query_time 开启 结合 mysqldumpslow / PMM

持久化：动态 SET GLOBAL 重启失效；写入 mysqld.cnf 后 systemctl restart mysql（维护窗口执行）。

19.4 备份层级、RPO/RTO 与恢复演练

备份类型 工具 优点 缺点
逻辑备份 mysqldump 简单、可跨版本 大库慢、恢复慢
物理备份 XtraBackup 快、适合大库 需专用工具与流程
Binlog 连续归档 点对点恢复 需与全量配合
  • RPO（可接受数据丢失时间）：Binlog 归档间隔 + sync_binlog 决定。
  • RTO（可接受恢复时间）：取决于备份体积与恢复脚本熟练度。

运维必做：季度 恢复演练（在隔离环境 mysql < backup.sql + mysqlbinlog 重放），验证备份可用而非「只备份不恢复」。

19.5 在线 DDL 与版本升级

大表 DDL（加列、加索引）在业务高峰可能长时间 MDL 堵表。生产可选：

  • MySQL 8.0 Inplace + ALGORITHM=INPLACE, LOCK=NONE（视 DDL 类型而定）。
  • 第三方：pt-online-schema-change（触发器）、gh-ost（基于 Binlog 的在线迁移），降低锁表风险。

5.7 → 8.0 升级检查清单：

  1. 全量逻辑 + 物理备份
  2. Staging 跑完应用回归与慢 SQL 对比
  3. 检查认证插件、保留字、GROUP BY 与 sql_mode
  4. 复制链先升从库、再切换主库
  5. 准备回滚方案（版本与备份）

19.7 生产变更 Checklist

变更前：

  • 有近期 可恢复备份
  • 在 Staging 或从库验证 SQL / DDL
  • 评估 锁、MDL、主从延迟 影响
  • 低峰窗口 + 可回滚方案 + 值班人

变更中：

  • 大表 UPDATE/DELETE 使用 LIMIT 分批
  • 长 DDL 使用在线工具或 LOCK=NONE
  • 监控 Threads_running、锁等待、复制延迟

变更后：

  • EXPLAIN / 业务抽样验证
  • 观察慢日志与错误率
  • 记录变更单与参数 diff

【规范】 数据库操作行为（JavaGuide 优化规范）：

  • 超 100 万行 的 INSERT/UPDATE/DELETE 必须 分批（主从延迟、binlog、大事务）。
  • 大表 DDL 优先 pt-osc / gh-ost（见 19.5）。
  • 程序账号 禁止 SUPER、禁止 DROP、单库单账号（见 第十三节）。

19.8 资深 DBA 仍须专深的方向

本篇覆盖 单机 + 常规主从 的开发与运维主干；以下方向需结合官方文档、厂商方案与专项实践：

方向 说明
物理备份与 PITR Percona XtraBackup、Binlog 链式恢复
高可用切换 MHA、Orchestrator、InnoDB Cluster、Keepalived + VIP
读写分离中间件 ProxySQL、MySQL Router
半同步 / 组复制 降低主库宕机丢数
分库分表 ShardingSphere、Vitess、按业务拆分
容器与 K8s StatefulSet、Operator、持久卷与备份
云 RDS / 托管库 参数组、只读实例、自动备份策略
性能诊断平台 PMM、Performance Schema 深度分析
审计与合规 Enterprise Audit、敏感数据脱敏
专项优化 内核参数、NVMe/IOPS、NUMA、thread_pool

这篇手册从 Obsidian 里的几条常用命令出发，已整合 架构原理、CRUD、性能、事务、监控、备份 与 JavaGuide 面经 考点（文中 【面经】 / 【规范】 标识）。建议：（一） 原理 → （二）～（七） 命令与运维 → 本篇 查漏与索引。
