6.5 命名与表设计规范【规范】

参考 JavaGuide · 高性能优化规范：

命名：

  • 库、表、字段 小写 + 下划线；见名知义，不宜超过 32 字符。
  • 避开 MySQL 保留字；临时表 tmp_YYYYMMDD，备份表 bak_ + 时间戳。
  • 禁止 预留字段；禁止开发/测试 直连生产；禁止在线对生产 压测。

表设计：

  • 统一 InnoDB + utf8mb4；所有表和字段 COMMENT。
  • 单表数据量建议 ≤ 500 万；更大用归档、分库分表；谨慎分区表，大数据优先物理分表。
  • 适度 反范式 冗余换查询，需保证一致性。
  • 经常一起用的列可放同表，减少 JOIN（权衡范式设计）。

7.7 性能要点（详见 第十一节）

写法 开销 更高效替代
SELECT * 多读列、网络与内存浪费 只查需要的列
LIKE '%词' 无法用 B+ 树索引，易全表扫描 全文索引、LIKE '词%'、ES

大表 COUNT(*) InnoDB 无索引时需扫大量行 缓存计数、汇总表、近似值

8.6 性能要点（详见 第十一节）

写法 开销 更高效替代
循环单行 INSERT 每条一次解析、日志、索引维护 多行批量 INSERT（8.2）
大批量导入 事务与 redo 压力大 LOAD DATA INFILE、分批提交
ON DUPLICATE KEY UPDATE 唯一键冲突检测 + 可能 UPDATE 明确场景再用；纯插入用普通 INSERT
INSERT ... SELECT 锁源表、占 undo/redo 低峰执行、控制批次、源表加合适索引

写法 开销 更高效替代
WHERE 无索引 全表扫描 + 逐行加锁 主键/索引定位

更新索引列 所有相关二级索引都要改 少改索引列；冗余列权衡
WHERE col LIKE '%x%' 同 SELECT 全表扫描 改写条件或走索引列

写法 开销 更高效替代
大批量 DELETE 长锁、undo 膨胀、空间不立刻释放 分批 DELETE ... LIMIT
