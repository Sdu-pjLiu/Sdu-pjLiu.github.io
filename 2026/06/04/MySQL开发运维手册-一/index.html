一、MySQL 架构、存储引擎与核心概念

读懂架构，才能理解后文「为什么这条 SQL 慢」「为什么 DELETE 后空间不释放」「主从为什么会延迟」。本节偏原理，命令速查从 第二节 开始。

1.1 MySQL 是什么：客户端 / 服务器

【面经】 关系型数据库（RDBMS）：数据存放在表中，表与表之间有一对一、一对多、多对多关系；通过 SQL 读写，主流引擎支持 ACID 事务。常见 RDBMS：MySQL、PostgreSQL、Oracle 等。

【面经】 MySQL 为何流行：开源生态成熟、InnoDB 事务与主从/读写分离方案完善、上手与运维成本相对低；默认端口 3306，默认引擎 InnoDB（5.5.5 起）。

MySQL 采用 C/S 架构：应用通过 mysql 客户端、JDBC、Python pymysql 等连接 mysqld 进程（服务器）。一个 mysqld 实例通常对应一个 datadir 目录，其下可挂载多个 database（schema）；库内是 table，表里是 row。

1.2 逻辑三层架构

层次 职责 开发与运维关注点
连接层 认证、权限、线程/连接管理 max_connections、连接池、短连接 vs 长连接
Server 层 解析 SQL、优化器、执行器、缓存、函数、跨引擎能力 EXPLAIN、慢 SQL、权限、Binlog（在此层）
存储引擎层 数据的物理读写、索引、事务、锁 InnoDB（默认）、表结构、行锁、Buffer Pool

MySQL 的 插件式存储引擎：CREATE TABLE ... ENGINE=InnoDB 可按表指定引擎；同一实例中不同表可用不同引擎（不推荐混用除非有明确理由）。

Server 层内部组件（一条 SQL 的执行路径）：

组件 作用
连接器 鉴权、权限、连接管理
分析器 词法 / 语法解析
优化器 选索引、JOIN 顺序，生成执行计划
执行器 调用存储引擎；执行前校验权限
查询缓存 MySQL 8.0 已移除；5.7 亦默认关闭

【面经】 查询缓存在表变更后易失效，写多读少场景弊大于利；8.0 起不要再考虑 query_cache。

1.5 存储引擎怎么选

引擎 事务 行锁 典型场景 备注
InnoDB 支持 支持 默认首选，OLTP MySQL 8.0 默认
MyISAM 不支持 表锁 只读归档、特殊遗留 崩溃恢复弱，不推荐新业务
MEMORY 不支持 表锁 临时会话数据 重启丢失
ARCHIVE 不支持 — 压缩归档 仅 INSERT/SELECT

实际开发：新业务统一 InnoDB；不要用 MyISAM 除非维护遗留系统。

【面经】 MyISAM 与 InnoDB 核心区别：

对比项 MyISAM InnoDB
事务 不支持 支持
锁 表锁 行锁（默认）
外键 不支持 支持（生产多在应用层约束）
崩溃恢复 弱 依赖 redo
MVCC 无 有
索引与数据 分离存储 聚簇索引即数据
缓存 Key Cache（仅索引） Buffer Pool（数据+索引）

【面经】 勿轻信「MyISAM 读更快」；《高性能 MySQL》指出：多数场景 InnoDB 更优，聚簇索引下尤其明显。5.5.5 起默认引擎为 InnoDB。

1.6 InnoDB 核心组件（与性能、运维直接相关）

组件 作用 运维 / 开发关联
Buffer Pool 缓存数据页，减少磁盘 I/O innodb_buffer_pool_size 常为内存 50%～70%
Redo Log 物理日志，崩溃后 前滚 已提交事务 写多、日志盘慢 → 吞吐下降
Undo Log 旧版本、回滚、MVCC 读 长事务 → undo 堆积 → 空间膨胀
Binlog 逻辑 SQL / 行变更，主从、备份 与 redo 两阶段提交 保证一致
Doublewrite 防止部分页写损坏 一般无需调，知悉即可
聚簇索引 数据按 主键 B+ 树 存储 主键选型影响插入与范围查询

1.7 三种日志：Redo、Undo、Binlog

日志 层级 类型 主要用途
Redo InnoDB 物理（页变更） 崩溃恢复，保证已提交事务持久化
Undo InnoDB 逻辑（反向） ROLLBACK、MVCC 读历史版本
Binlog Server 逻辑（SQL/行） 复制、按时间点恢复

开发启示：任何提交成功的写，通常都会进 Binlog（从库要 replay）；长事务未提交则 Undo 保留旧版本，阻塞 purge。

【面经】 Redo 与 Binlog 的两阶段提交（2PC）：开启 Binlog 时，事务提交大致为「InnoDB prepare（redo 落盘）→ 写 Binlog → InnoDB commit」。二者共同保证崩溃恢复后主从、点对点恢复一致；因此主库常设 innodb_flush_log_at_trx_commit=1 与 sync_binlog=1（见 （八）19.3）。

binlog_format 复制内容 注意
ROW（生产推荐） 行变更 日志量大，但少歧义、主从一致性好
STATEMENT 原始 SQL 可能因 UDF、非确定性函数导致主从不一致
MIXED 混合 由 Server 按语句选择

1.8 索引：B+ 树与聚簇 / 二级索引

InnoDB 索引结构为 B+ 树（多路平衡树，叶子链表便于范围扫描）：

  • 聚簇索引（主键）：叶子节点存 完整行；表即索引，必须有主键（无则隐藏 row_id）。
  • 二级索引：叶子存 索引列 + 主键值；查询常需 回表（再查聚簇索引）。

选型建议：

主键选择 插入表现 说明
BIGINT AUTO_INCREMENT 顺序插入，页分裂少 推荐
随机 UUID 随机插入，页分裂多 可用有序 UUID（如 UUID v7）
业务字段 视情况 需短、不变、单列

索引越多，INSERT/UPDATE/DELETE 越慢（每个索引都要维护）。

【面经】 为什么用 B+ 树而不是 Hash / B 树？

  • B+ 树「矮胖」，千万行约 3～4 层 I/O，且叶子链表便于 范围查询。
  • Hash 不支持范围、排序、联合索引最左前缀；MEMORY 引擎可用 Hash，InnoDB 通用索引用 B+ 树。
  • 相对 B 树：B+ 非叶只存键、扇出更大；任意查询路径长度稳定。

【面经】 覆盖索引：索引已含 SELECT 所需全部列 → Extra: Using index，避免回表。

【面经】 联合索引 (a,b,c) 的最左前缀：可用 (a)、(a,b)、(a,b,c)；以下仅 class 条件 不走 (name,class) 索引：

【面经】 a=1 AND c=1 走 (a,b,c) 时只用 a 列，再过滤 c；b=1 AND a=1 AND c=1 优化器可能重排为 a,b,c 顺序。

索引失效常见原因：

原因 示例
违反最左前缀 见上
列上函数 / 计算 WHERE YEAR(create_time)=2026
隐式类型转换 WHERE id = '123'（id 为整型）
前导 % LIKE LIKE '%abc'
OR 一侧无索引 a=1 OR b=2 且 b 无索引
大 IN / 大结果集 优化器可能改全表扫

索引设计规范（JavaGuide 优化规范）：

  • 单表索引 建议 ≤ 5；优先 联合索引，勿每列单独建索引。
  • 必须有主键；推荐 自增 BIGINT，避免 UUID/MD5 随机主键。
  • 区分度高的列放联合索引 最左；范围条件列放 右侧。
  • OLTP 慎用全文索引；外键 建议在业务代码保证一致性。

【面经】 适合建索引的列：WHERE / JOIN / ORDER BY / GROUP BY 常用列；高区分度列；外键关联列（不必建 DB 外键约束）。
【面经】 索引未必更快：表很小、结果集占表很大比例、统计信息过时 — 优化器可能选全表扫。

1.9 事务、ACID 与 MVCC

ACID：

特性 含义 InnoDB 实现要点
Atomicity 原子性 全成或全不成 Undo 回滚
Consistency 一致性 约束不被破坏 引擎 + 应用共同保证
Isolation 隔离性 并发互不干扰 MVCC + 锁
Durability 持久性 提交后不丢 Redo + Binlog

【面经】 ACID 中 C（一致性） 常由 应用 + 事务机制 共同保证，并非仅由数据库单独实现。

MVCC（多版本并发读）：普通 SELECT 读 快照（历史版本来自 Undo），一般 不加行锁；SELECT ... FOR UPDATE 才当前读并加锁。

【面经】 并发事务导致的四类问题：

问题 含义
脏读 读到别的事务 未提交 的数据
不可重复读 同事务内两次读 同一行，结果不同（他人 UPDATE）
幻读 同事务内两次 范围读，第二次行数变多/少（他人 INSERT）
丢失更新 两事务同时改同一数据，后写覆盖先写

【面经】 幻读 vs 不可重复读：前者强调 新增/消失的行；后者强调 同一行内容被改。InnoDB 对 INSERT 防幻读依赖 Next-Key Lock。

隔离级别（transaction_isolation，MySQL 8.0）：

级别 脏读 不可重复读 幻读 常用场景
READ UNCOMMITTED 可能 可能 可能 极少用
READ COMMITTED 否 可能 可能 Oracle 默认；部分 ORM
REPEATABLE READ 否 否 否* MySQL 默认
SERIALIZABLE 否 否 否 性能差，锁多

*InnoDB 在 RR 下：一致性读（普通 SELECT）在同一事务内复用 Read View，可避免幻读；当前读（FOR UPDATE 等）靠 Next-Key Lock 防插入。上表按 SQL 标准定义；MySQL/InnoDB 实际表现以引擎为准。

【面经】 隔离级别实现：RC/RR 主要靠 MVCC（undo + Read View）；SERIALIZABLE 靠锁；RR 下 当前读（FOR UPDATE）仍用 Next-Key Lock 防幻读。

实务命令见 第十二节。

1.10 锁：行锁、间隙锁与 MDL

锁类型 典型触发 影响
行锁（Record Lock） UPDATE/DELETE 命中行 阻塞其他写同一行
间隙锁（Gap Lock） 默认 RR 下范围条件（RC 通常无间隙锁） 阻塞范围内插入
Next-Key Lock 行锁 + 间隙锁 RR 默认，防幻读
MDL（元数据锁） 访问表 / ALTER TABLE DDL 与 DML 互斥，易「堵表」
意向锁 IS/IX 行锁前在表级标记 快速判断能否对表加表锁

【面经】 InnoDB 行锁加在索引上：UPDATE/DELETE 的 WHERE 未命中索引 时，可能对扫描到的所有行加锁（甚至接近锁表）。

【面经】 快照读 vs 当前读：

  • 快照读：普通 SELECT，走 MVCC，一般不加锁。
  • 当前读：SELECT ... FOR UPDATE/FOR SHARE、INSERT/UPDATE/DELETE，加 S/X 锁。

【面经】 自增锁：AUTO_INCREMENT 插入涉及 innodb_autoinc_lock_mode（8.0 默认 2 交错模式，并发更好）；Statement 格式 binlog 主从需注意插入顺序。

排查：SHOW ENGINE INNODB STATUS\G、第十五节、第十八节 锁等待查询。

1.11 复制与高可用（架构视角）

经典 异步主从：

角色 开发注意 运维注意
主库 写、强一致读 Binlog 磁盘、sync_binlog
从库 延迟读可能旧 Seconds_Behind_Source、只读账号
半同步 / GTID 减少丢数 gtid_mode、故障切换

读写分离：写主、读从；对一致性敏感读应走主库或等同步。

【面经】 架构层优化（索引/SQL 优化仍不够时）：读写分离 → 分库分表 → 冷热分离 → Redis 缓存热点；后者引入复杂度，非必要不优先。
【面经】 深度分页：用 WHERE id > ? ORDER BY id LIMIT n 代替大 OFFSET（见 11.2）。

1.14 库表设计基础（开发应知）

常用数据类型（InnoDB + utf8mb4 前提下）：

类型 适用场景 注意
TINYINT / INT / BIGINT 状态、计数、ID 金额、哈希 ID 用 BIGINT；布尔可用 TINYINT(1)
DECIMAL(p,s) 金额、精确小数 不要用 FLOAT / DOUBLE 存钱
VARCHAR(n) 变长字符串 n 按实际需要，过大浪费内存与缓存
CHAR(n) 定长、短码 如固定长度编码
DATETIME / TIMESTAMP 时间 TIMESTAMP 有时区转换，存储 UTC；TIMESTAMP 上限约 2038-01-19（2038 问题），DATETIME 范围更大
JSON 半结构化（8.0 常用） 可建函数索引；不宜替代所有关系建模
BLOB / TEXT 大文本、二进制 大行影响 Buffer Pool；考虑对象存储 + URL

约束与关系：

设计原则（精简）：

  • 每张 InnoDB 表应有 明确主键（推荐自增 BIGINT 或有序 ID）。
  • 业务唯一性用 UNIQUE 约束，不要只靠应用层判断。
  • 字符集统一 utf8mb4，排序规则常用 utf8mb4_unicode_ci 或 _0900_ai_ci（8.0）。
  • 字段尽量 NOT NULL + 默认值，简化索引与优化器行为。
  • 大表预留 归档 / 分区 / 分表 策略，不要等亿级才想。

【面经】 CHAR vs VARCHAR：CHAR 定长（不足补空格），适合 MD5、身份证等固定长度；VARCHAR 变长，适合昵称、标题。VARCHAR(100) 与 VARCHAR(10) 磁盘占用相近，但排序/内存可能按声明长度分配，100 更耗内存。

【面经】 UNSIGNED：非负整数上限翻倍，自增 ID、状态码优先使用 INT UNSIGNED / BIGINT UNSIGNED。

【面经】 手机号用 VARCHAR，不用 BIGINT：前导零、+86、加密密文无法用数字类型表达。

【规范】 不要用字符串存日期；禁止 FLOAT/DOUBLE 存金额；慎用 ENUM（改值需 DDL，推荐 TINYINT + 字典）；TEXT/BLOB 大字段单独表或避免 SELECT *；禁止在库存图片/大文件（存 URL）。

【规范】 单表建议 ≤ 500 万行（非硬限）；表与字段加 COMMENT；关联列 类型必须一致，避免隐式转换导致索引失效。

库表命名与设计规范见 第六节 6.5。

1.15 MySQL 5.7 与 8.0 关键差异

维度 5.7 8.0
默认认证插件 mysql_native_password caching_sha2_password
默认字符集 常为 latin1（安装后需改） 默认 utf8mb4
查询缓存 有（易成瓶颈，常关） 已移除
窗口函数 / CTE 无 ROW_NUMBER()、WITH 等
不可见索引 无 CREATE INDEX ... INVISIBLE 便于验证
降序索引 伪降序 真降序索引
复制术语 SHOW SLAVE STATUS 推荐 SHOW REPLICA STATUS
Redo 日志 ib_logfile* #innodb_redo 目录

升级 5.7 → 8.0 前务必：全量备份、在 staging 跑应用回归、检查 sql_mode、认证插件、保留字冲突（如 RANK、GROUPS）。详见 19.5。

1.16 sql_mode 与严格模式（开发必查）

sql_mode 决定「非法/宽松数据」是否被拒绝，直接影响 INSERT/UPDATE 是否静默截断或报错。

常见模式 作用
STRICT_TRANS_TABLES 事务表非法值报错（推荐开启）
ONLY_FULL_GROUP_BY GROUP BY 列与 SELECT 非聚合列一致（8.0 默认常含）
NO_ZERO_DATE / NO_ZERO_IN_DATE 拒绝零日期
ERROR_FOR_DIVISION_BY_ZERO 除零报错而非 NULL

实践：新环境对齐 Staging 与生产 的 sql_mode；从 5.7 升到 8.0 后，应用 SQL 可能因 ONLY_FULL_GROUP_BY 报错，需提前在测试库验证。
