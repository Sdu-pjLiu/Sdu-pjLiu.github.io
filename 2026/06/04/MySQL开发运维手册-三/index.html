十一、CRUD 性能开销与优化实践

本节说明 SELECT / INSERT / UPDATE / DELETE 在 InnoDB 下大致「贵在哪里」，以及日常可落地的更高效写法。MySQL 的读写大多走 Buffer Pool；未命中缓存会触发磁盘 I/O。写操作除数据页外，还要写 redo log、undo log（支持事务回滚与 MVCC），若开启 binlog 还有同步复制开销。

11.1 四种操作开销对比（概览）

操作 主要成本 锁 是否写 Binlog 空间是否立刻回收
SELECT CPU + 缓冲池读；可能磁盘读 一致性读一般不加行锁（LOCK IN SHARE MODE / FOR UPDATE 除外） 否 —
INSERT 数据页写入 + 索引维护 + redo/undo/binlog 插入意向锁、行锁 是 —
UPDATE 先定位行 + 写新版本 + 改索引 + redo/undo/binlog 行锁（范围大时可能间隙锁） 是 否（留空洞/碎片）
DELETE 标记删除 + 改索引 + undo + binlog 行锁 是 否（需 OPTIMIZE 或 DDL 整理）
TRUNCATE DDL，重建表更快 表级 MDL 是 是（整表重置）

经验排序（同量级数据、同等索引条件下）：SELECT（只读）< INSERT 批量 ≈ 单行 UPDATE by PK < 大批量 UPDATE/DELETE << 无索引条件的 UPDATE/DELETE 全表扫。

11.2 SELECT（读）——开销与优化

开销从哪来

  1. 解析与优化：生成执行计划；复杂 JOIN / 子查询 CPU 更高。
  2. 访问路径：走主键/二级索引 range/ref 成本低；ALL（全表扫描） 随表线性增长。
  3. 回表：二级索引查到的列不全时，需回主键再取行（随机 I/O）。
  4. 排序与临时表：ORDER BY / GROUP BY 无合适索引时可能 filesort、Using temporary。
  5. 深分页：LIMIT offset, n 仍要扫描 offset + n 行再丢弃前 offset 行。
  6. 返回列过多：SELECT * 增大网络传输与应用内存。

低效示例 → 更高效写法

（1）避免 SELECT *，用覆盖索引减少回表

（2）深分页改为 keyset（游标）分页

（3）避免 leading wildcard 的 LIKE

（4）大表 COUNT：区分精确与近似

（5）JOIN 与 IN：保证关联列有索引

（6）只读分析用只读账号 + 从库，避免拖慢主库（见 第十三节、（八）快速索引）。

11.3 INSERT（写）——开销与优化

开销从哪来

  1. 聚簇索引（主键）插入：顺序主键（自增）通常比随机 UUID 主键 页分裂少。
  2. 二级索引：每插一行，每个二级索引都要维护。
  3. 日志：每条写入产生 redo；事务未提交前还有 undo。
  4. Binlog / 主从：同步复制时，从库 replay 同样语句。
  5. 唯一约束：INSERT IGNORE / ON DUPLICATE KEY UPDATE 需先查唯一索引。

低效示例 → 更高效写法

（1）避免应用层循环单行插入

（2）超大批量：事务分批 + LOAD DATA

（3）主键设计：自增 / 有序 ID 优于随机 UUID

（4）INSERT ... SELECT 控制批次与锁

11.4 UPDATE（改）——开销与优化

开销从哪来

  1. 定位行：与 SELECT 相同；无索引则全表扫描。
  2. InnoDB 更新：实际多为 插入新版本行 + 标记旧版本删除（MVCC），并非原地覆盖。
  3. 索引维护：凡 被更新且出现在索引中的列 都会触发索引修改；更新 WHERE 未用到的列若不在索引中则相对便宜。
  4. 锁：命中行加 X 锁；范围条件在 RR 隔离级别下可能有 间隙锁。
  5. 大批量：长事务 → undo 表空间膨胀、锁等待、主从延迟。

低效示例 → 更高效写法

（1）务必用索引列（最好是主键）定位

（2）大批量分批 + 短事务

（3）避免「值未变仍 UPDATE」

（4）多表关联更新：确保驱动表走索引

11.5 DELETE（删）——开销与优化

开销从哪来

  1. 逐行删除：每行 undo + 索引条目删除 + binlog。
  2. 空间：InnoDB 不会立刻缩小文件，留下 碎片；需 OPTIMIZE TABLE 或重建表（生产慎用）。
  3. 锁与长事务：与大批量 UPDATE 类似。
  4. TRUNCATE：本质是 DDL，会隐式提交；执行完成后不能像 DELETE 一样回滚（即便在 BEGIN 之后执行），且无 WHERE。

DELETE vs TRUNCATE vs DROP

命令 适用 速度 可带 WHERE 自增重置 触发器
DELETE 按条件删 慢 是 否 会
TRUNCATE 清空整表 快 否 是 否（InnoDB 不触发 DELETE 触发器）
DROP TABLE 删表结构 最快 — — —

低效示例 → 更高效写法

（1）历史数据：分批 DELETE 或分区 DROP

（2）归档再删，减轻主表体积

（3）清表优先 TRUNCATE 而非 DELETE

11.6 写操作通用优化清单

  1. 先 EXPLAIN，再上线：读与「带 WHERE 的写」都适用。
  2. 索引不是越多越好：写越多索引，INSERT/UPDATE/DELETE 越慢。
  3. 控制事务粒度：批处理可以，但单事务不宜过大（建议万级行内分批提交）。
  4. 低峰做 DDL 与大清理：ALTER、OPTIMIZE、大批量 DML。
  5. 读写分离：报表、对账类 SELECT 走从库。
  6. 监控：Slow_queries、锁等待（第十五节、（八）快速索引）、主从 Seconds_Behind_Source。
  7. 字符集与行宽：更宽的行 = 更多 I/O；utf8mb4 必要列即可。

11.7 用 EXPLAIN 判断「贵不贵」

Extra 字段常见警示：

Extra 含义 建议
Using filesort 额外排序 为 ORDER BY 建索引或改写 SQL
Using temporary 建临时表 优化 GROUP BY / DISTINCT
Using where 存储引擎层过滤 结合 rows 看是否扫太多行
Using index 覆盖索引 读路径较优

11.8 SQL 编写规范【规范】

规范 说明
复杂运算放应用层 数据库专注存储与过滤
禁止 SELECT * 指定字段；利于覆盖索引（【面经】 非 WHERE 导致失效，但浪费 IO）
禁止无列名 INSERT INSERT INTO t(c1,c2) VALUES (...)
PreparedStatement 防 SQL 注入、可复用计划
避免隐式转换 见 1.8 索引失效
子查询改 JOIN 大结果集子查询易临时表
JOIN 表数 建议 ≤ 5
同列 OR 改 IN IN 列表不宜过大（约 500 内视情况）
禁止 ORDER BY RAND() 易全表加载内存
WHERE 不对列做函数 WHERE create_time >= '2026-01-01' AND create_time < '2026-01-02'
UNION ALL 代替 UNION 避免去重临时表
拆分复杂大 SQL 单 SQL 单 CPU，可并行多个小 SQL
程序账号不跨库 利于分库与最小权限

11.9 性能优化体系【面经】

优化顺序（点 → 线 → 面，JavaGuide 面经）：

  1. 慢 SQL 定位：慢日志、performance_schema（第十五节）。
  2. EXPLAIN / EXPLAIN ANALYZE：看 type、key、rows、Extra（见下节、第十六节）。
  3. 索引 + SQL + 表结构（第一节 1.8、本节 11.2～11.8）。
  4. 架构层（最后手段）：读写分离、分库分表、冷热分离、Redis 缓存。
  5. 连接池与硬件：池大小、Buffer Pool、SSD（（八）19.3）。

【面经】 SELECT * 不会直接「导致索引失效」，但妨碍覆盖索引、增大网络与内存开销。
【面经】 索引不是越多越好：写路径维护每个索引；单表建议 ≤ 5 个。

12.2 隔离级别

级别 设置语句 说明
读未提交 READ UNCOMMITTED 几乎不用
读已提交 READ COMMITTED 每条语句新 Read View；8.0 默认不启用间隙锁，并发通常更好

12.4 死锁与锁等待

InnoDB 会自动 回滚死锁中代价较小的事务；应用应捕获死锁错误并重试。

开发建议：

  • 多表更新时 固定加锁顺序，减少死锁。
  • 缩短事务：锁在事务内停留时间越短越好。
  • 避免在事务里做 RPC / 外部 HTTP，防止长事务。
